Заливка изменений на сервер

Понятное дело что все изменения сделаны на локальной машине и их необходимо залить (запушить) на сервер. Это делается в несколько этапов т.к. все заливаемое на сервер должно быть маркировано определенным коммитом.
1) Добавить изменения, которые сделали локально 

git add .
2) Подписать изменения и сделать коммит 
git commit -m "commit text"
3) Закачать (запушить) последний коммит на сервер в свою ветку (бранч) 
git push origin branch_name

Создание веток, переход на ветку и изменение кода в ветке

Групповой проект подразумевает что каждый из участников группы будет работать в своей ветке (бранч), а тим лид будет по необхобимости объединять ветки в цельный проект (мержить).

Начнем с создания веток для каждого участника группы. Для создания ветки используется следующая команда: 

git checkout -b branch_name
Для переключения на другую ветку используется такая-же команда. Другими словами, создание и переход на ветку выполняется одной и той-же командой (кроме атрибута "-b"), все зависит от того есть такая ветка уже или нет. Объединение веток (мерж)

Объединение веток выполняется командой: 

git merge branch_name
Логика процесса объединения следующая:
  1. переходим на ветку в которую хотим слить данные из другой ветки
  2. выполняем команду слияния и указываем имя ветки, которую хотим слить в базовую ветку

Планирование

При планировании важно учесть следующие моменты:

  • Работая в команде вам необходимо распределить обязанности между участниками группы.
  • Для того чтобы планы не забылись можно использовать например файл Readme вашего проекта.
  • Разделите обязанности так чтобы каждый участник работал со своим отдельным файлом в своей ветке, так будет проще объединять (мержить) отделльные ветки в один проект.
  • Важно определить не только обязанности, но и время на выполнение каждой части.
  • Так-же будет правильным подходом назначить встречи (в онлайне) для обсуждения текущего прогресса, что-бы в случае "запар" у одного участника команда могла вовремя подхватить и помочь.
  • При встречах неплохо проводить командную "Ревью" кода (Об этом подробнее в след разделе).
  • При завершении какого либо этапа разработки важно что-бы тим лид объединял все в единый проект (мержил).

Ревью кода

Ревью кода это Анализ кода друг друга — один из главных путей взаимодействия между разработчиками во время разработки. Преимущества:

  • Взаимобучение
  • Простота передачи кода другому разработчику
  • Помощь разработчику если вдруг произошла "запара"
  • Налаживание взаимодействия между разработчиками
Недостатки:
  • время, однако благодаря взаимообучению и налаженному взаимодействию, потраченное время плавно переходит в повышение професионализма разработчиков и соответственно ускорение разработки в будущем

Комментарии — пояснения к исходному тексту программы, находящиеся непосредственно внутри комментируемого кода. Синтаксис комментариев определяется языком программирования. С точки зрения компилятора или интерпретатора, комментарии — часть текста программы, не влияющая на её семантику. Комментарии не оказывают никакого влияния на результат компиляции программы или её интерпретацию. Помимо исходных текстов программ, комментарии также применяются в языках разметки и языках описания.

Другими словами комментарии это часть кода, невидимая компилятору (в нашем случае браузеру) и никак не влияющая на функционал программы. Используется для пояснения функционала другим разработчикам. 

                
                <!-- Комментарий в HTML -->
                /* Коментарий в CSS */
